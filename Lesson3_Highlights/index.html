Version Control System (VCS)

Version control is a repository system which records all changes that have been made to a project. The system works for any kind of project including documentation, design or development that benefit from backing up and reverting to previous statuses.

The version control functions save and a lot of time, and energy by giving the ‘undo’ options in the process of development. Version control system notes who changes the file when the changes occurred, and what the change is.

VCS's Evolution

Version control programs have evolved through time. Initially, Local Version Control Systems (LVCS) was developed. These were simple databases system that copies all changes made to files, and then recreate the files as it was on a specified date.

Centralized Version Control Systems (CVCS) were developed to respond to the need for being able to collaborate on a single project. These programs placed their different versioned files in a centralized database and allow different programmers to check out the files. Although this system is an excellent tool for administration, it has a single point failure as a major drawback. When the server is down, all people working on the database will be unable to save their work.

Distribution Version Control System (DVCS)addressed the issue of single point of failure in CVCS. This system lets clients check out files with their full backup history. If the master file in the repository is damaged for some reason, the checkout files could be used to replace it. This makes distributed version control systems more powerful than both local and centralized version control systems.

Git: Distributed Version Control System

Git is a Distributed Version Control System (DVCS) developed by Linux development community in 2005 having the goals of speed, simplicity, strong support for non-linear development, fully distributed and ability to handle large projects. Git differs from other version control systems in a way it sees data. Other systems see data as files and record the change made to those files over time. Gits take a snapshot of each version and save or stores them only when there is a change or a difference. It considers its data like a streamline of snapshots. Git has some more benefits over the other systems

Git Benefits

Local Operation: local computers usually have all the necessary files and resources required to run Git. It is possible to work and commit offline and push later on when a connection is available.

Integrity: Git will detect when a change was made to a content or when files get corrupted in the transition process.

Only add data: Git generally only add data to a database and it is always possible to undo, which facilitates room for experimentation.

Three states: Git files has three states

  1. committed: when the data is stored in the database.
  2. modified: when a change has been made to a file but not yet stored in the database
  3. staged: when the modified files are marked to be committed

The three status of files in a repository dictates Git workflow which is modified, add and commit.

Git Repository Local and Remote

Having a basic understanding of command lines (navigation and changing file system) helps to have a good grip of Git commands and work in a Git repository. There are two ways to get a Git repositor

  1. Turning local directory to Git repository
  2. Cloning an existing repository from somewhere

When working in the existing Git repository, initializing the Git is the first step. The git command for doing so is git init and it should be run inside the working directory. This command creates a folder inside the working directory and placed all necessary files.

Running git Clone copies an existing Git repository i.e. every version of every file in the history will be cloned. Git Clone command has an argument that specifies which directory to clone and we give name of our target directory

There are two kinds of files in our working directory: tracked and untracked. Tracked files are the ones that are modified, staged or committed and as a result, included in the latest version of Gits snapshot. Untracked files are those files, which are not included in the latest snapshot. When untracked files are modified, Git notes the change and record their snapshot.

Git Commands The Three Basics

Here are some of the basic Git commands that we use in Git’s workflow which involves modifying, staging and committing files to a Git Repository.

Checking a Status: git status is used to check the status of files. This command checks whether there are untracked files and display the list. It also shows on which branch we are currently on .git status -s or git status –short gives a simplified result. Git status list out the just name of files that are modified, however, git diff gives a much more detailed account of the change in the file.

Tracking New files: git add is used to look for new files that are not tracked. This command put the files in the staging area, where they will be ready to be stored in the repository. This command takes the name of the file or directory that is to be tracked.

Modified File: git add is also used when we want to stage modified files. These files have been tracked or added before, but their new status is not captured until we run git add. this command helps them be captured in their latest version.

Committing Changes: git commit is the command used to finally store the file to a repository. It will store all the files that are staged not necessarily modified. We use git rm and git mv to remove and move riles respectively
